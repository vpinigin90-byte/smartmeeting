## 1. Обзор проекта

### 1.1 Цель проекта

Создать веб-приложение для автоматизации бронирования демонстрационных встреч Scroll Tool, которое позволит потенциальным клиентам самостоятельно записаться на демо конструктора онлайн-курсов, минуя ручную координацию через менеджера.

### 1.2 Текущая ситуация (AS-IS)

**Существующий процесс:**
1. Менеджер по продажам обзванивает лидов из CRM
2. Уточняет актуальность покупки Scroll Tool
3. Предлагает демонстрационную встречу
4. Смотрит 2 календаря: свой + календарь ведущего (методист или владелец)
5. Переписка/перезвоны с клиентом для согласования времени
6. Менеджер вручную создает встречу, добавляет всех участников, ссылку МТС-Линк, комментарии

**Проблемы:**
- Долгий процесс согласования (переписка туда-сюда)
- Менеджер тратит время на координацию вместо продаж
- Клиент ждет ответа, может потерять интерес
- Ручная работа - риск ошибок и забывчивости

### 1.3 Желаемая ситуация (TO-BE)

**Новый процесс:**
1. Клиент видит форму на сайте Scroll Tool
2. Выбирает удобное время из доступных слотов (без менеджера)
3. Заполняет данные и бронирует встречу
4. Автоматически создается встреча в МТС-Линк + добавляется в календари команды
5. Клиент и команда получают уведомления

**Преимущества:**
- Мгновенная запись 24/7 без участия менеджера
- Клиент сам выбирает удобное время
- Менеджер фокусируется на продажах, а не на координации
- Профессиональный имидж компании

## 2. Функциональные требования

### 2.1 Основные функции (MVP)

#### F1: Просмотр доступного времени
**Описание:** Клиент видит только свободные слоты, когда хотя бы один человек из команды доступен

**Требования:**
- Отображение календаря на текущий и будущие месяцы
- Рабочее время: **Понедельник-Пятница, 10:00-19:00 МСК**
- Слоты с шагом **30 минут** (10:00, 10:30, 11:00, 11:30...)
- Блокировка прошедших дат и времени
- Выходные (суббота, воскресенье) не отображаются

**Логика доступности слота:**
- Слот свободен = хотя бы один из команды свободен (Методист 1 ИЛИ Методист 2 ИЛИ Владелец)
- Система проверяет календари Mail.ru всех 4 человек
- Клиент НЕ видит, кто конкретно проведет встречу
- Клиент НЕ видит названия задач в занятых слотах (приватность)

**Критерии приемки:**
- [ ] Календарь показывает только Пн-Пт
- [ ] Время отображается с 10:00 до 19:00
- [ ] Слоты по 30 минут
- [ ] Прошедшие даты недоступны
- [ ] Занятые слоты не отображаются вообще (не показываем "занято")

#### F2: Выбор длительности встречи
**Описание:** Клиент выбирает продолжительность демонстрации

**Требования:**
- 3 варианта длительности:
  - **30 минут** (экспресс-демо)
  - **45 минут** (стандартное демо)
  - **60 минут** (полное демо с вопросами) - **по умолчанию**
- Выбор длительности влияет на доступные слоты

**Критерии приемки:**
- [ ] Клиент видит 3 кнопки выбора длительности
- [ ] 60 минут выбрано по умолчанию
- [ ] Слоты пересчитываются при смене длительности

#### F3: Форма бронирования
**Описание:** Сбор информации о клиенте и цели встречи

**Обязательные поля:**
- Имя
- Фамилия
- Email
- Телефон

**Опциональные поля:**
- Telegram (ник) - для уведомлений в Telegram
- Дополнительные участники (список email через запятую или построчно)
- Комментарий/вопросы для обсуждения на встрече

**Критерии приемки:**
- [ ] Валидация email формата
- [ ] Валидация телефона (российский формат)
- [ ] Возможность добавить несколько участников
- [ ] Поле комментариев с подсказкой "Укажите вопросы, которые хотите обсудить"
- [ ] Кнопка "Назад" на каждом шаге

#### F4: Автоматическое распределение ведущего
**Описание:** Система автоматически назначает ответственного за проведение демо

**Алгоритм (Round-Robin с приоритетом):**

1. **Проверка методистов:**
   - Если Методист 1 свободен И его очередь → назначить Методиста 1
   - Если Методист 2 свободен И его очередь → назначить Методиста 2
   - Если оба свободны → назначить того, чья очередь (чередование 1-2-1-2...)

2. **Если оба методиста заняты:**
   - Проверить владельца продукта
   - Если свободен → назначить владельца
   - Если тоже занят → слот не доступен для бронирования

3. **Менеджер:**
   - Добавляется ВСЕГДА во все встречи автоматически
   - Для видимости в его календаре
   - Физически присутствует на 1 из параллельных (на свое усмотрение)

**Критерии приемки:**
- [ ] Методисты чередуются равномерно (50/50 нагрузка)
- [ ] Владелец назначается только когда оба методиста заняты
- [ ] Менеджер добавлен во все встречи
- [ ] В календаре клиента не видно, кто конкретно назначен

#### F5: Создание видеовстречи в МТС-Линк
**Описание:** Автоматическое создание видеоконференции

**Требования:**
- Создание встречи в МТС-Линк при бронировании
- Название: "Демо Scroll Tool - [Имя Фамилия клиента]"
- Длительность: выбранная клиентом (30/45/60 минут)
- Участники:
  - Назначенный ведущий (методист/владелец)
  - Менеджер по продажам
  - Клиент (email)
  - Дополнительные участники клиента (если указаны)

**Критерии приемки:**
- [ ] Встреча создана в МТС-Линк
- [ ] Все участники добавлены
- [ ] Ссылка для подключения сгенерирована
- [ ] Название содержит имя клиента

#### F6: Синхронизация с календарями Mail.ru
**Описание:** Автоматическое добавление события в календари команды через CalDAV

**Требования:**
- Подключение к 4 календарям Mail.ru:
  - v.pinigin@scroll-tool.ru (владелец)
  - [email методиста 1]
  - [email методиста 2]
  - [email менеджера]

- Создание события в календарях:
  - Название: "Демо Scroll Tool - [Имя Фамилия]"
  - Описание: Ссылка МТС-Линк + комментарий клиента + участники
  - Время: выбранное клиентом
  - Участники: все участники встречи

**Критерии приемки:**
- [ ] События созданы в календарях назначенного ведущего и менеджера
- [ ] Ссылка МТС-Линк в описании события
- [ ] Комментарий клиента виден в описании
- [ ] Участники добавлены в событие

#### F7: Email уведомления
**Описание:** Календарное приглашение клиенту

**Требования:**
- Отправка .ics файла (календарное приглашение) на email клиента
- Клиент может принять встречу в своем календаре (Google/Outlook/Mail.ru/любой)
- Содержание:
  - Дата и время
  - Длительность
  - Ссылка МТС-Линк
  - Имя ведущего (методист/владелец)
  - Контакты для связи

**Критерии приемки:**
- [ ] Клиент получает .ics файл
- [ ] Приглашение открывается в любом календаре
- [ ] Ссылка МТС-Линк кликабельна
- [ ] Письмо приходит в течение 1 минуты

#### F8: Telegram уведомления

**F8.1: Уведомление клиенту**
**Требования:**
- Если клиент указал Telegram ник → отправить сообщение через бота
- Содержание:
  - Подтверждение бронирования
  - Дата и время
  - Ссылка МТС-Линк
  - Контакты для связи

**F8.2: Уведомления команде (отдельный бот для команды)**

**При создании встречи:**
- Уведомление назначенному ведущему (методист/владелец)
- Уведомление менеджеру
- Информация: клиент, дата, время, вопросы для обсуждения

**Напоминания перед встречей:**
- За 1 час: "Встреча с [Имя] через 1 час"
- За 15 минут: "Встреча с [Имя] через 15 минут"
- За 5 минут: "Встреча с [Имя] через 5 минут"  
- За 1 минуту: "Встреча с [Имя] начинается!"

**После встречи (только менеджеру):**
- Чек-лист задач:
  - [ ] Получить запись встречи
  - [ ] Отправить запись клиенту
  - [ ] Написать письменное резюме встречи
  - [ ] Отправить план следующих шагов
  - [ ] Внести договоренности в CRM

**Критерии приемки:**
- [ ] Клиент получает Telegram уведомление (если указал ник)
- [ ] Команда получает уведомление о новой встрече
- [ ] Напоминания приходят в указанное время
- [ ] Менеджер получает чек-лист после встречи
- [ ] Бот работает надежно (99%+ доставка)

#### F9: Подтверждение бронирования
**Описание:** Экран успешного бронирования

**Требования:**
- Красивая страница подтверждения
- Информация:
  - "Встреча успешно забронирована!"
  - Дата и время
  - Длительность
  - Ведущий (имя, но не email)
  - Ссылка МТС-Линк
  - "Вы получите приглашение на email и в Telegram"
- Кнопка "Забронировать еще встречу"

**Критерии приемки:**
- [ ] Экран подтверждения отображается
- [ ] Все детали встречи показаны
- [ ] Ссылка МТС-Линк кликабельна
- [ ] Можно вернуться к бронированию

### 3.3 База данных (Supabase)

**Таблица: bookings**
```sql
CREATE TABLE bookings (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  
  -- Время встречи
  date DATE NOT NULL,
  time TIME NOT NULL,
  duration INTEGER NOT NULL, -- 30, 45 или 60 минут
  
  -- Данные клиента
  client_first_name VARCHAR(255) NOT NULL,
  client_last_name VARCHAR(255) NOT NULL,
  client_email VARCHAR(255) NOT NULL,
  client_phone VARCHAR(50) NOT NULL,
  client_telegram VARCHAR(255), -- необязательно
  client_additional_participants TEXT[], -- массив email
  client_questions TEXT, -- комментарий/вопросы
  
  -- Назначенные участники
  assigned_host VARCHAR(255) NOT NULL, -- email ведущего (методист/владелец)
  assigned_host_role VARCHAR(50) NOT NULL, -- 'methodist1', 'methodist2', 'owner'
  manager_email VARCHAR(255) NOT NULL, -- всегда одинаковый
  
  -- Интеграции
  mts_link_meeting_id VARCHAR(255),
  mts_link_join_url TEXT,
  calendar_event_ids JSONB, -- {owner: 'id', manager: 'id', methodist1: 'id'}
  
  -- Статус
  status VARCHAR(50) DEFAULT 'confirmed', -- confirmed, cancelled, completed
  
  -- Метаданные
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  
  -- Уведомления
  email_sent BOOLEAN DEFAULT FALSE,
  telegram_sent BOOLEAN DEFAULT FALSE
);

-- Индексы
CREATE INDEX idx_bookings_date ON bookings(date);
CREATE INDEX idx_bookings_status ON bookings(status);
CREATE INDEX idx_bookings_assigned_host ON bookings(assigned_host);
CREATE INDEX idx_bookings_client_email ON bookings(client_email);
```

**Таблица: team_rotation**
```sql
CREATE TABLE team_rotation (
  id SERIAL PRIMARY KEY,
  last_assigned VARCHAR(50) NOT NULL, -- 'methodist1' или 'methodist2'
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Инициализация
INSERT INTO team_rotation (last_assigned) VALUES ('methodist2');
-- Начинаем с methodist2, чтобы первая встреча пошла на methodist1
```

### 3.4 Безопасность

**API:**
- Rate limiting: 10 запросов/минуту с одного IP
- CORS: только с домена booking.scroll-tool.ru
- HTTPS обязателен
- Валидация всех входных данных

**Данные:**
- API ключи только в переменных окружения
- Пароли календарей зашифрованы
- Telegram токены защищены
- База данных: Row Level Security (RLS) в Supabase

**Защита от спама:**
- reCAPTCHA v3 на форме бронирования (опционально)
- Rate limiting на бронирования: 3 встречи/час с одного IP
- Email верификация через домен

### 3.5 Производительность

**Целевые метрики:**
- Время загрузки страницы: < 2 секунд
- Время отклика API: < 500ms
- Проверка доступности слотов: < 300ms
- Создание бронирования: < 2 секунд

**Оптимизации:**
- Кэширование доступных слотов (5 минут)
- Параллельные запросы к календарям (Promise.all)
- CDN для статических ресурсов
- Compression (gzip/brotli)

**Масштабируемость:**

## 8. Метрики успеха

### 8.1 Ключевые метрики (KPI)

**Эффективность автоматизации:**
- **Время на запись 1 встречи:**
  - Сейчас (AS-IS): 10-15 минут (переписка с менеджером)
  - Цель (TO-BE): < 2 минут (самостоятельная запись)
  - **Экономия:** ~13 минут на встречу × 4-7 встреч/неделю = **52-91 минуты/неделю**

- **Количество бронирований через форму:**
  - Месяц 1: > 5 бронирований (тестовый период)
  - Месяц 2: > 15 бронирований
  - Месяц 3+: > 20 бронирований/месяц

**Конверсия:**
- Посетители страницы → Начало бронирования: > 40%
- Начало бронирования → Завершение: > 70%
- Общая конверсия: > 28%

**Загрузка команды:**
- Методист 1: 40-50% встреч
- Методист 2: 40-50% встреч
- Владелец: < 20% встреч (только когда оба методиста заняты)

**Техническая производительность:**
- Uptime: > 99.5%
- Время загрузки страницы: < 2 секунды
- Успешность бронирований: > 98% (без технических ошибок)
- Доставка уведомлений: > 99%

**Пользовательский опыт:**
- Время на полное бронирование: < 2 минуты
- Мобильные пользователи: > 40% от общего трафика
- Повторные бронирования: > 5% (клиенты возвращаются)
